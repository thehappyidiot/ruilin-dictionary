# Personal Dictionary <!-- Think of a better name -->

This project is personalized Dictionary webapp. 

## The Idea
Many groups of people (romantic relationships, friends, family, etc.) have their own set of words that only they use.
This set of words changes over time.
It would be cool to keep of track of one's own dictionary.
It can sent as a gift, or just used as a reference.

## Features
- For now, it is just a webapp for one couple to keep track of their vocabulary
- There should be a way to enter your own words, but ONLY IF YOU'RE THE ADMIN. See security
- Because, it's designed as a gift, it should look fun and playful. NOT professional and dull as an actual dictionary

## UI

### Word page
- Search field to search for words
- Fields
  - Word: The actual word itself
  - Type: Eg. noun, verb, adjective, etc
  - Meaning: meaning of the word
  - Sentence: word used in a sentence
  - Origin: where this word/phrase came from
  - Not to be confused with: A simple list of strings (no links, plain text)
- A button for showing a random word
- If you're the admin, a button to Edit the listing.
- Upon opening, the user is greeted with a random word

### Entering a word:
- Basically the same fields as the word page, but to enter data, instead
- Basic validation - word, type, meaning and sentence are mandatory
- Origin and "not to be confused with" are optional
- A button to submit

## Architecture

### Backend
- Go
- Postgres
- SQLc
- Goose

### Database Schema
- Words Table
  - ID: autogenerated primary key
  - word: the word, string (could be a phrase, so should be long)
  - type: string
  - meaning: long string
  - sentence: long string
  - origin: long string
  - confused_with: array of strings (plain text entries)

### Deployment / Hosting
- Running on a self hosted server via Docker container
- Distributed through cloudflare tunnels

### Admin Verification / Security
- Store a bcrypt password hash in environment variables (`ADMIN_PASSWORD_HASH`)
- Store a strong session secret in environment variables (`SESSION_SECRET`)
- If a user goes to `/admin`, ask for the password and compare it against the bcrypt hash
- If successful, set admin state in a secure gorilla session cookie
- Session cookie settings: `HttpOnly`, `Secure` in production, `SameSite=Lax`, finite max age
- Protect admin POST endpoints with CSRF tokens
- Add login attempt throttling (per-IP + global) and small backoff
- Use generic failure messages for login
- ALWAYS verify every admin request on the backend (UI visibility is convenience only)
